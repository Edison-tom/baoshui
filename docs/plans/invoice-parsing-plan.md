# 发票原件解析方案设计

## 问题现状

导入发票文件夹（PDF/OFD/图片）时，没有任何数据被识别。

当前 ImportPanel 已经能正确检测文件类型（显示为 `invoice_original`），但点击"确认导入"后实际没有解析发票内容——只存了空的 `invoices[]`。

## 依赖状态

| 依赖 | 版本 | 用途 | 是否已安装 |
|------|------|------|-----------|
| jszip | ^3.10.1 | 解压 OFD 文件包 | ✅ |
| pdfjs-dist | ^6.1.200 | 提取 PDF 文本 | ✅ |
| tesseract.js | ^7.0.0 | 图片 OCR 识别 | ✅ |
| ofd-parser.ts | — | OFD XML 提取（已有骨架） | ✅ 需补全 |

三个依赖都已安装，缺失的是**调用这些依赖的解析逻辑**。

## 解析方案

### 1. OFD 发票（电子税务局下载的主力格式）

```
OFD 文件结构：
  ├── OFD.xml（包描述）
  ├── Doc_0/
  │   ├── Document.xml（文档结构）
  │   └── Pages/
  │       └── Page_0/
  │           └── Content.xml（发票内容 XML）
  └── DocumentRes/（字体、图片等资源）
```

**方案**：用 JSZip 解压 `.ofd` 文件 → 读取 `Doc_0/Pages/Page_0/Content.xml` → 用 DOMParser 解析 XML 提取字段（已实现 `parseOfdInvoiceXML`）→ 转换为 `InvoiceItem`。

### 2. PDF 发票（电子税务局导出格式、邮箱收到的电子发票）

**方案**：用 `pdfjs-dist` 加载 PDF → 提取文本内容 → 正则匹配关键字段（发票号码、金额、日期、购销方名称、税额）。

### 3. 图片发票（手机拍照、扫描件）

**方案**：用 `tesseract.js` 进行 OCR → 提取识别文本 → 同 PDF 的正则匹配逻辑。

### 4. 通用正则提取器

PDF 和图片 OCR 提取的都是**纯文本**，需要一个通用函数从文本中匹配发票字段：

| 字段 | 正则/匹配方式 |
|------|-------------|
| 发票号码 | `发票号码[：:]\s*(\S+)` |
| 发票代码 | `发票代码[：:]\s*(\S+)` |
| 开票日期 | `开票日期[：:]\s*(\S+)` |
| 金额（不含税） | `(?:金额\|不含税金额)[：:]\s*([\d,.]+)` |
| 税额 | `(?:税额\|增值税额)[：:]\s*([\d,.]+)` |
| 价税合计 | `(?:价税合计\|含税金额)[：:]\s*([\d,.]+)` |
| 销售方 | `(?:销售方\|销方)[名称：:]*\s*(\S+)` |
| 购买方 | `(?:购买方\|购方)[名称：:]*\s*(\S+)` |

### 5. 整合到 ImportPanel

1. 拖入文件时，对于 `invoice_original` 类型（OFD/PDF/图片），调用对应解析器
2. 解析结果存入 `fileStoreRef` 的 `invoices[]`
3. 显示解析出的条目数（例如"识别 3 张发票"）
4. 对识别失败的显示具体原因

## 注意点

- **CORS/跨域**：都在浏览器端运行，不涉及后端 API，纯本地解析
- **PDF 多页**：有些发票是扫描件合并成 PDF 的，需要遍历所有页面
- **OCR 速度**：中文字体 OCR 较慢（每张 1-3 秒），大文件夹会卡，需显示加载状态
- **JSZip 动态 import**：按需加载，不增大初始包体积
- **正则鲁棒性**：不同电子发票平台（航信、百旺、税务UKey）文本格式略有差异，需兼容
